# 攻擊向量目錄

> 系統化的攻擊面向，確保「攻得全面」——不是隨便攻，而是按面向逐項攻。

依命題的 type 從下表選相關類別。每類有幾個攻擊模式 + steel-manned（最強化）範例。

## 通用面向（決策/架構/計畫/策略/投資）

### 假設（ASSUMPTIONS）— 高風險，失效前常隱形
- **隱藏假設揭露**：揪出沒明說、卻撐著整個命題的信念。例：「你假設使用者願意多付 20%，但沒有定價研究支撐，市場上類似功能都免費。」
- **承重挑戰**：命題成敗繫於某假設，但信心不足。例：「整個 ROI 押在 30% 採用率，業界基準是 10-15%；若 15%，NPV 轉負。」
- **假設堆疊**：成功要 A 且 B 且 C 都成立，各 70%，合起來只剩 34%，且無 fallback。
- **時效衰減**：假設依賴「現況持續」，但條件會變，有沒有 hedge？

### 替代方案 / 機會成本（ALTERNATIVES）— 決策審查核心
- **do-nothing baseline**：跟「什麼都不做」比，這決策真的更好嗎？例：「你算了新方案 ROI 正，但沒跟維持現狀比；現狀零成本、零風險，新方案要扣掉切換成本後才有意義。」
- **只比較了一個選項**：是否只評估了偏好方案、沒列出 2-3 個對照案？例：「你直接選了自建，沒比 SaaS/外包/混合；缺對照組等於沒做決策只做了背書。」
- **被排除選項沒被公平評估**：淘汰的選項是因為真的差，還是因為沒認真看？例：「方案 B 被一句『太複雜』帶過，但它正好避開了方案 A 的承重假設。」
- **機會成本**：投這個就投不了別的，被犧牲的次佳用途值多少？

### 可逆性 / 鎖定（REVERSIBILITY）— 決策審查核心
- **單向門 vs 雙向門**：這是 Type-1（難回頭，要慎）還是 Type-2（可逆，該快）決策？把可逆決策當不可逆會拖慢，把不可逆當可逆會出大事。例：「資料庫選型是單向門，遷移成本極高，卻用了雙向門的草率程度拍板。」
- **退場成本 / lock-in**：若決策錯了，回頭代價多大？vendor lock-in、資料綁死、合約鎖死、團隊技能沉沒。例：「選了專屬 API，三年後想換要重寫 40% 程式碼，退場成本沒被計入。」
- **錯誤代價非對稱**：做錯 vs 不做的代價是否嚴重不對稱？可逆且下檔有限 → 該快試；不可逆且下檔巨大 → 該慢。

### 依賴（DEPENDENCIES）
- 外部依賴消失（上游 API/服務/marketplace 下線、套件棄更）。
- 內部依賴循環、單點故障、版本鎖死。

### 邊界案例（EDGE_CASES）
- 空輸入/極大輸入/同名碰撞/併發/跨平台差異（如 Windows 路徑、權限）。

### 擴展性（SCALABILITY）
- 量大/量小時崩潰；成長後維護成本爆炸。

### 維運（OPERATIONAL）
- 日常執行卡點、人為流程斷鏈、需要某人手動的步驟容易被忘。

### 競爭 / 市場 / 經濟（COMPETITIVE / ECONOMIC）
- 競爭者反制、市場條件變化、財務可行性、現金流。
- 例（市場/單位經濟學）：「你押 LTV/CAC=3，但獲取成本逐季升、留存未驗證；業界早期 SaaS 中位 LTV/CAC 約 1.5，現金流燒到 break-even 常比預估晚 12-18 個月。」
- 例（競爭/護城河）：「護城河靠功能領先 6 個月，但無專利/網路效應，head-start 在資本充足對手下平均 9-12 個月被抹平。」

### 時機（TEMPORAL）
- 時程無 buffer、依賴的事件延遲、視窗稍縱即逝。

### 人 / 組織（ORGANIZATIONAL）
- 需要人改變習慣（歷史變革 70% 失敗率）、文化阻力、權責不清。

### 二階後果 / 系統效應（SECOND_ORDER / SYSTEMIC）— 高價值，常被「會成功」遮蔽
- **誘因扭曲（Goodhart / Campbell）**：指標一旦被當目標就失效。例：「上了採用率 KPI 後，團隊改去刷註冊數而非真實留存，數字漂亮但商業價值反降。」
- **非預期後果 / 成功反而帶來的問題**：方案達成主目標，卻誘發沒料到的副作用。例：「自動化跑通後沒人再懂底層流程；半年後出事無人能 debug，反而比手動更脆。」
- **回饋迴路**：行為改變系統、系統再回頭放大行為（正回饋失控 / 負回饋抵銷努力）。例：「降價搶市佔引發競爭者跟進降價，最終全產業毛利下殺、你佔不到便宜。」
- **外部性**：成本被轉嫁給沒參與決策的第三方（其他團隊 / 下游 / 未來的自己）。例：「為趕版砍測試，技術債轉嫁給維運團隊，三個月後 oncall 成本爆炸。」

### 可證偽性 / Pre-mortem（FALSIFIABILITY）— 逼命題給出「怎麼知道自己錯了」
- **反證條件缺失**：命題沒說明什麼觀測會推翻它；無法證偽的命題本身即弱點。
- **失敗的領先指標**：假設它已失敗，回推為什麼（Klein pre-mortem），要求列出失敗的早期可觀測訊號。
- **kill-criteria 缺失**：採 MITIGATE/ACCEPT 時須給出「達到什麼門檻就觸發 kill switch / 撤回」的明確判準，與藍方 MITIGATE 的 kill switch 及殘餘風險對接。例：「你說失敗就停損，但沒定義『失敗』長什麼樣——採用率低於多少、連續幾週無回購算失敗？沒有 kill-criteria 的 kill switch 永遠不會被扣下。」

## 安全面向（security）— 對應 SKILL 的 security type
> 攻擊面/漏洞/依賴/維運 可複用通用面向的 DEPENDENCIES、OPERATIONAL；以下補安全姿態專屬向量。
- **信任邊界與輸入驗證**：跨信任邊界的輸入沒驗證、過度信任上游資料。
- **認證 / 授權繞過（authn/authz）**：未驗證身份即放行、權限檢查可被繞過、IDOR。
- **權限提升**：低權限拿到高權限路徑、預設過寬權限。
- **機密與金鑰管理**：明文存密鑰、token 進 git/log、輪替缺失。
- **供應鏈 / 依賴漏洞**：第三方套件已知 CVE、安裝腳本任意執行、typosquatting。
- **DoS / 資源耗盡**：無上限輸入、放大攻擊、無 rate limit。
- **稽核與可觀測性**：關鍵操作無紀錄、出事無法追溯。
- **威脅建模（STRIDE / 攻擊樹）**：用 STRIDE 逐項過、畫攻擊樹找最省力的攻破路徑。

## 程式碼面向（code）— 本專案實戰

- **崩潰**：null ref、無效輸入、資源耗盡、競態、溢位。要可重現。
- **資安/注入**：injection、path traversal、控制字元注入、資料外洩、不安全反序列化。要 PoC。
- **跨平台**：Windows cmd.exe 不認 Unix 語法（`2>/dev/null`、`&&`）、檔案 symlink 需權限、CRLF。（本 session 多次踩到。）
- **錯誤處理一致性**：JSON.parse 沒包 try/catch、fail-safe 把無法確認當成 0（如 rev-list NaN 當已 push）。
- **邏輯/狀態**：dirty 狀態機沒閉合、entry 缺漏靜默降級、原子寫缺失。
- **路徑逃逸**：name 含 `../` 逃出目錄；要白名單驗證。
- **效能/複雜度**：熱路徑 O(n²)/O(n³) 退化、N+1 query、不必要的全量載入/重複計算（要能指出退化的輸入量級）。
- **相容性破壞**：公開 API/函式簽章/DB schema 破壞、序列化格式（JSON 欄位/enum 值）變更未版本化、下游消費者靜默壞掉。
- **測試缺口**：關鍵路徑零覆蓋、難以測（強耦合/無注入點）、改動未補對應測試。

## plugin/skill/agent 配置面向（config）— 本專案實戰

- **觸發詞衝突/失靈**：description 觸發詞與別的 skill 撞、或全英文使用者中文觸發不到。
- **權限過寬**：plugin 拿到不需要的工具、hook 能跑任意指令。
- **路徑逃逸 / 敏感外洩**：registry 存絕對路徑進 git、token 沒被 .gitignore 排除、adoptedFrom 沒脫敏。
- **互動指令邊界**：Claude 代執行了不該代執行的 `/plugin`；或文件叫使用者貼根本不存在的指令（如 `/plugin update`）。
- **覆蓋風險**：狀態存 plugin 內、被更新覆蓋（vs 該存家目錄）。
- **prompt/agent 設計**：觸發詞誤命中/漏命中、skill 間指令衝突、不可信輸入經上下文流入造成 prompt injection、模式切換邊界、output schema 漂移、工具/上下文假設未成立。
- **設計/可用性**：指引自相矛盾或模式/分支不可判定（使用者不知該走哪條）、對宣稱支援的用例淪為形式（確認劇場、走編輯既有非真新增）、預設值與聲稱不符、認知負荷過高、與其他元件職責重疊。

## 文件一致性面向（docs）— 本專案實戰

- **文件聲稱 vs 實際**：教學說有某功能/某 hook，實際不存在（吹牛）。
- **殘留過時**：移除某指令/欄位後，文件還提它（如殘留 `:new`、`externalCandidates`）。
- **版號/清單不符**：plugin.json bump 了，教學版號沒同步；scripts/plugin 清單與實際不符。
- **斷鏈**：目錄錨點指向不存在的章節。
- **指令可貼性**：文件示範 `/plugin uninstall && install`（slash 不能 && 串接）。

## 嚴重度基準

| 嚴重度 | 準則 | 例 |
|------|------|----|
| CRITICAL | 承重 + 信心 < 50% | 「市場存在」零驗證、會丟資料的狀態機漏洞 |
| HIGH | 承重 或 信心 < 50% | 時程無 buffer、跨磁碟整個功能壞 |
| MEDIUM | 影響效率非可行性 | UX 不一致、潛在 robustness（當前不觸發） |
| LOW | 錯了可補救 | 文件措辭、註解殘留 |

> 由高至低依序套用，命中即定級——先測 CRITICAL（承重且信心<50%），不符再測 HIGH（承重或信心<50%），再 MEDIUM、LOW。CRITICAL/HIGH 準則重疊區一律取較嚴重者。
