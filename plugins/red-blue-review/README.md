# red-blue-review — 中文紅藍對抗

> 在現實攻擊你之前，先自己把弱點攻出來。

對任何命題（決策 / 架構 / 計畫 / 策略 / 程式碼 / plugin 配置 / 文件）做**對抗式壓力測試**：
**Red 用最強攻擊找弱點 → Blue 防禦／強化 → 迴圈到收斂**，產出一個經得起攻擊的強化版命題 + go/no-go 建議。

這是**對抗式論證**（挑戰假設、戳破盲點），不是效能／負載壓測，也不是漏洞掃描器。

## 安裝

透過 fulin plugin marketplace 安裝：

```
/plugin install red-blue-review
```

裝好後以觸發詞自然喚起，無需額外設定。

## 怎麼用

### 觸發詞

說出以下任一即觸發：「紅藍對抗」、「攻擊這個」、「找弱點」、「壓力測試」、「挑戰這個決策」、「這個方案會出什麼問題」、「幫我戳破」、「魔鬼代言人」、「盲點」、「站得住腳嗎」、「有什麼沒想到的風險」、「反方論點」、「red team」、「stress test」、「poke holes」、「devil's advocate」等。

### 要指定的兩件事

啟動後 skill 會（從上下文或詢問）確認：

1. **標的**：要對抗什麼？（一段程式碼 / 一個架構決策 / 一份計畫 / 一個 plugin 配置…）
2. **面向 type**：決定攻擊角度，可多選。

| type | 用在 | 預設攻擊面向 |
|------|------|------|
| `decision` | 決策審查 | 假設、替代方案、可逆性、後果、時機 |
| `architecture` | 架構強化 | 擴展性、安全、依賴、維運、邊界案例 |
| `plan` | 計畫審查 | 可行性、資源、時程、依賴、風險 |
| `strategy` | 策略驗證 | 競爭、市場、執行、依賴、時程 |
| `investment` | 投資評估 | 經濟、市場、執行、競爭、假設 |
| `security` | 安全姿態 | 攻擊面、漏洞、依賴、維運 |
| `code` | 程式碼審核 | 崩潰、注入／資安、邏輯錯誤、邊界、跨平台、錯誤處理 |
| `config` | plugin/skill/agent 配置稽核 | 觸發詞衝突、權限過寬、路徑逃逸、敏感外洩、與實況不符 |
| `docs` | 文件一致性 | 文件聲稱 vs 實際、殘留過時、版號／清單不符、斷鏈 |

> 上表為常見起點，非窮舉。流程／用人／產品／合規等其他命題，挑最接近的 type 並借用通用面向即可（`ASSUMPTIONS`、`DEPENDENCIES`、`SECOND_ORDER`、`FALSIFIABILITY` 等對任何命題皆適用）。完整攻擊向量見 `skills/red-blue-review/references/attack-catalog.md`。

### 兩種模式

| | 實作模式（預設） | 分析模式 |
|---|---|---|
| 適用 | `code`/`config`/`docs`（有實檔可改） | decision/architecture/plan/strategy/investment，或只要報告不要自動修 |
| 流程 | 紅攻 → 藍守 → **自動修** → 持續迴圈 | 紅攻 → 藍守後**停，給確認清單** → 確認後才修 → 續迴圈 |
| 停下問人 | 僅**高風險改**（刪檔 / 跨 repo / 改設定檔 / 不可逆）才停 | 首輪一定停給清單；後續修正**鎖定第一輪確認範圍**（檔案集＋顧慮集），超出先提醒 |

**怎麼選**：要實際修且面向可改 → 實作模式；要先人工把關、或命題沒實檔可改 → 分析模式。意圖不明且面向可改 → 走實作模式，高風險改檔時自會停下問。

實作模式標的大時可接 Workflow spawn 獨立 agent 平行稽核（紅方找 finding → 藍方獨立讀實檔驗真偽），詳見 `skills/red-blue-review/references/workflow-pattern.md`。

## 收斂機制：迴圈到攻不破

兩模式共用核心條件：紅攻 → 藍守 →（修）→ 再跑一輪……**直到某一輪紅方產生 0 個新的真弱點才停**（唯一輪 0 新即收斂）。

幾道關鍵閘門防止迴圈空轉或淪為「確認劇場」：

- **新真弱點的定義**：藍方獨立驗證為真（非假陽性）、去重後非變體、且嚴重度 ≥ MEDIUM。
- **修正門檻**：只修藍方確認為真且 ≥ MEDIUM 的弱點；**LOW 一律只記錄不修**——斷掉「為續圈硬找雞毛蒜皮」的誘因。
- **去重基準**：對「所有提過的攻擊（含已剔除的假陽性）」比，只看底層顧慮——否則假陽性每輪重現、永不收斂。
- **面向覆蓋閘**：宣告收斂前，該 type 的每個預設攻擊面向都須至少認真攻過一輪。
- **硬上限**：預設 5 輪（`round_cap`），達上限強制停並回報殘餘弱點。

可調參數（`round_cap` / `dry_rounds` / `all_addressed`）見 `skills/red-blue-review/references/convergence.md`。

最終產出包含：攻擊覆蓋三態、確認弱點＋處置（HARDEN/MITIGATE/ACCEPT）、剔除的假陽性、ACCEPT 的殘餘風險、LOW 級清單、強化後命題、以及 **GO / BLOCK** 建議。

## 何時用 / 何時不用

**該用**：高風險決策、架構或策略定案前、計畫審查、發布前審核、安全姿態、投資評估、plugin/skill/agent 配置稽核、文件與實況一致性。

**不該用**（反模式）：低風險瑣事（直接做）、緊急救火（先滅火再復盤）、已定案無法回頭、想法還在成形期（過早攻擊）、走形式的確認劇場（沒有真心對抗就別做）。

### 與其他工具的分工

| 工具 | 做的事 | 何時選它 |
|------|------|------|
| **red-blue-review** | 對抗式論證：找最強攻擊 → 強化命題 | 要戳破假設、找盲點、壓力測試一個命題 |
| `security-review` | 純掃 PR 安全漏洞 | 只要逐項漏洞掃描 |
| `code-reviewer` | 純程式碼規範審查（DDD/CQRS、AI 痕跡、CS1591…） | 只要依架構／規範逐條審查 |

> code 面向若使用者意圖偏向逐項漏洞掃描而非思辨對抗，本 skill 會先反問確認再決定是否轉介。

## 重要限制（誠實告知）

- steel-manning 是為了**求真不是求贏**——攻擊強化到最強、藍方守得住，才證明命題真的穩。
- 嚴重度與 go/no-go 是**輔助判斷**，最終決策仍由使用者拍板。
- **對話內單實例自我紅藍時，藍方獨立性有限**（同一模型推理鏈，confirmation/anchoring 偏誤無法靠「叫自己當藍方」根除）。高風險命題建議走實作模式 spawn 獨立 agent，或請第二個實例／人複核。
