# 測試計畫設計（方法論）

QA Agent 在 Phase 1 產出測試計畫的通用規範。**不假設任何特定 UI 框架、元件庫或後端**——
所有技術棧專屬的雷與領域知識都在 `knowledge/`，不在這裡。

## 0. 設計前置：盤點既有測試資產（先於覆蓋矩陣）

動筆寫 TC **之前**，先盤點目標專案 runner 內**既有的測試自動化資產**——page object / helper / fixtures /
相似的既有 TC。兩種情況：

- **（A）已有既有資產** → 能復用就優先復用，新 TC 與既有資產的風格、命名對齊。
  - **怎麼盤（手段）**：枚舉 runner 既有測試——`pytest --collect-only`（或讀 tests/ 目錄與描述性測試函式名）+ 讀 xfail / skip marker。
    這能盤出「**已寫的**測試」供復用對齊，但**回答不了「該做卻沒人寫的情境」**（見 §0.5）——枚舉只能列出存在的測試，不存在的測試在 collect-only 輸出裡是隱形的。「要沿用還是新建」的完整答案需搭配 §0.5 的情境覆蓋索引。
- **（B）greenfield（專案尚無 runner / 測試資產）** → 建立第一個：
  - **不要自己 mkdir / 自己判斷落點**：一律先跑 `qa-flow.sh bootstrap`（見 `SKILL.md` Phase 2），
    它鎖 `CLAUDE_PROJECT_DIR`、發 `ASSET: none` + `ACTION-REQUIRED: ask-user-install` 訊號、確保 catalog.md 存在。
  - **落地目錄與命名（由腳本鎖死）**：測試放在**你起 session 的那個資料夾**底下的 `tests/e2e/`（檔名 `test_<feature>.py`），
    **禁止自行鑽進子專案目錄**——這條規範由 `qa-flow.sh` 用 `CLAUDE_PROJECT_DIR` 機械執行，AI 想鑽也鑽不進去
    （歷史踩雷：主 Agent 看到子目錄有既有 JS 專案就鑽進去出 JS）。
    > **要讓 e2e 進某個專案 repo 版控，正確作法是「在那個專案資料夾裡起 session」**，而非從上層資料夾鑽進去——e2e 的測試棧/語言不一定等於該專案語言，落點由「session 起在哪」決定、不由 skill 寫死絕對結構。
  - **runner 固定優先 pytest**：使用者同意後 `qa-flow.sh scaffold <feature> pytest` 建骨架，安裝指令
    （`pip install pytest-playwright && playwright install chromium`）由腳本印出、**使用者自行執行（腳本不代裝）**；
    僅使用者明確不同意裝 Python 才退而 `scaffold <feature> playwright-js`。
  - 專案專屬值（port / login / BASE_URL）仍委派目標專案 `CLAUDE.md`，不寫進方法論。

> 與 §1「讀範圍內程式碼」不同：§1 讀的是**被測功能的產品原始碼**（API/前端/store）；本節盤的是**既有的測試碼**。
> 漏了這步，QA Agent 會在不知道既有資產存在的情況下從零設計，產出與既有回歸資產重疊或風格不一致的計畫，
> 弱化 codify 閉環的一致性。

## 0.5 持久化情境覆蓋索引（**必做**，跨功能、含未覆蓋、邊 codify 邊登記）

§0 的 runner 枚舉只能回答「**已寫了哪些測試**」，回答不了「**該做卻沒人寫的情境**（❌未覆蓋）」——
而後者正是 R3「這情境要沿用還是新建」的答案來源。`pytest --collect-only` 是對「已存在測試」的枚舉，
一個「使用者會做、但還沒人寫測試」的情境在它的輸出裡**結構性隱形**（無法枚舉不存在的東西）。

> **兩個不重疊的 SSOT，不是漂移**：runner 是「**已寫測試**」的單一真相來源；情境覆蓋索引是「**應測情境＋各自覆蓋狀態（含 ❌）**」的單一真相來源。
> 兩者語意不重疊——索引補的是 runner 結構上沒有的「差集語意（哪些情境還沒被覆蓋）」。漂移屬「維護紀律」問題，不是「不該有索引」的理由。

因此：**從第一條測試起就維護一份跨功能、持久化的情境覆蓋索引**，邊 codify 邊累積（呼應 codify 閉環）。
**這份索引固定落在 session 起始目錄的 `catalog.md`（一份總表、跨功能累積），由 `qa-flow.sh catalog <情境> <函式> <狀態> <模組>` 機械回填**（以測試函式為主鍵 update/append，不靠 AI 記得手寫，見 `SKILL.md` Phase 2 step 6）。
**這份索引與 `conftest.py` / `test_*.py` / report 同屬必產物——只做 pytest 那套卻漏掉索引，等於漏了 R3 的承載結構（這是常見漏法，務必一併產出）。**
**預設用 markdown 表**（四欄如下）——**這份索引的讀者是「人」**（QA Agent 盤點、開發者查「這情境測了沒」），不是機器；md 表是人類掃得動、好維護、進 git 可 diff 的形式。
> **逃生門（窄）**：只有當專案已有「對人更好讀」的既有索引（如專案 wiki、既有覆蓋報表）才沿用之；**禁止用純機器格式（xml / json）當人類索引**——那等於沒給人看，違背本索引存在的目的。
> （對比：`conftest.py` / `test_*.py` / `report.xml` 讀者是機器、格式由 runner 決定；本索引讀者是人、故釘 md。**格式服從讀者**。）

最小四欄骨架（欄位語意中立、不綁專案值，但形式預設 md 表）：

| 欄位 | 內容 |
|------|------|
| 白話業務情境 | 人類可讀的「使用者會做什麼」（**非函式名**） |
| 對應測試函式 | 函式名，或 `—`（尚無對應測試） |
| 覆蓋狀態 | ✅完整（操作＋讀回） / ⚠️部分（附缺口原因，如「只測守門、正向待補」） / ❌未覆蓋 |
| 業務模組分類 | 所屬模組（可對到頁面 / API / 角色維度） |

> **`—` ＋ ❌未覆蓋 那一列，正是 R3「沿用還是新建」的答案來源**，也是 collect-only 結構上產不出的那一列。

**從第一條測試就建、不要等規模大了回溯補**：邊 codify 邊登記是「幾乎零增量成本」的累積；等到逾百函式平坦清單已退化成「憑印象掃」（正是 §1 與 §0 警告要避免的）才想回補，得回溯整理上百個既有測試的情境/覆蓋狀態，成本爆炸。延後 = 利滾利的技術債。

**防漂移紀律（不是 open question，就靠 codify 閉環同步）**：
- **新增覆蓋** → codify 一條測試後，於閉環末端回填該情境列（見 `SKILL.md` Phase 2 step 6）。
- **改/刪測試** → 同步更新或移除對應情境列的測試函式欄與覆蓋狀態（測試刪了該列退回 ❌未覆蓋）。
- 索引與 runner 同源於同一次 codify 動作，不另開維護時機——這就是「不漂移」的機制。

## 1. 覆蓋矩陣（必做，先於寫 TC）

讀完範圍內的程式碼後，**先**把所有「狀態維度」窮舉成一張表，每格至少對應一條 TC。
禁止憑印象直接跳到寫 TC。

需窮舉的維度（依本次功能取用）：

- **Status / 狀態**：相關狀態的每個值與轉移（A→B）
- **分支條件**：每個 if / switch / 條件渲染（含 happy + 失敗分支）
- **角色 / 權限**：每種有差異的使用者情境
- **視圖 / Tab 狀態**：每個 tab、每種列表狀態（空 / 單筆 / 多筆 / 跨視圖轉移）
- **資料邊界**：空 / 最小 / 最大 / 特殊字元 / null

| 維度 | 值 / 分支 | 對應 TC | 備註 |
|------|----------|---------|------|
| 狀態 | 待處理 | TC-003 | |
| 角色 | 一般使用者 | TC-010 | |

**任一格沒有對應 TC = 計畫不完整，不得交付。**

## 2. 可追溯性（必做）

- 每一條驗收條件 / 需求點，至少對應一條 TC。
- 本次 diff 改動到的每個分支、每個新增/修改的互動，至少對應一條 TC。
- 計畫末尾附「需求 ↔ TC 對照表」；不測的需求點明確標「不測 + 理由」。

## 3. 測試案例格式

```
TC-{編號}
功能：{名稱}
前置條件：{完整進入路徑、必要的初始狀態}
操作步驟：
  1. {具體操作：點哪個元素 / 填什麼值 / 在哪個欄位} → 【證據】{這步要驗證什麼}
     預期結果：{可由其結構化證據獨立判定的結果}
  2. ...
需檢查項目：
  - [ ] 無執行期錯誤（console 無紅色錯誤）
  - [ ] 預期的狀態/資料變化發生
  - [ ] {其他特定檢查}
```

**關鍵規則：**
- 每個操作步驟標 **【證據】** 並說明驗證重點（對映一個 critical point + 一行 assert）。
- 每個步驟附**預期結果**，且寫成「看一個證據就能判定 pass/fail」的形式。
- 操作描述要具體，前置條件含完整進入路徑。

## 4. 必測 Checklist（設計時逐類涵蓋）

通用類別。實際檢查點依本次功能取用；技術棧專屬的「怎麼驗」放 `knowledge/`。

- **功能正確性**：主流程可完成、操作後狀態正確改變、改變後正確顯示、查詢/篩選回正確結果、成功/失敗訊息正確。
- **驗證邏輯**：必填留空觸發驗證訊息且禁止送出、格式錯誤正確提示。
- **UI 行為**：載入無執行期錯誤、loading 正確開關、文案/i18n 無顯示 key 名、互動元素 enabled/disabled 正確。
- **資料流程**：資料正確流向其來源（若有外部請求/持久化，驗其結果；細節見 `knowledge/`）、送出格式正確、回來的資料正確渲染。
- **多步驟 / 分頁 / wizard 表單**：每完成一步須以快照或讀回**確認該步資料已進入 state**，不可一次填到底才送出——否則中途某步未被收進 state 時，最終 payload 會缺漏該欄位且難定位是哪一步掉的。TC 須拆成逐步 state 確認，不寫成單一條「填完即送」。
- **錯誤路徑**：來源回錯誤時 UI 正確顯示、不白屏、不卡 loading；業務失敗不誤判成功；重複/並發操作不產生重複或互蓋；驗證失敗時在前端就擋下。
- **Edge case（預設必測，跳過寫理由）**：空狀態、重複操作、極長文字/特殊字元、邊界值。
- **權限（預設必測，跳過寫理由）**：有權限可執行、無權限元素不顯示、跨情境可見範圍正確。

> 若本次確無權限差異或 Edge case，須在計畫明寫「不適用 + 理由」，不得默默省略。

## 5. 測試資料建立規範（原則）

需建立測試資料時，**每個可填欄位都要填合理且真實的值**，不得偷懶留空。

**原則性原因**：留空的非必填欄位會在資料層沉澱「壞值」（空字串 / null / 型別預設值 / 邊界日期），
後續流程讀到壞值可能觸發例外、序列化錯誤或渲染異常，**讓真 bug 被遮蔽**，或讓二次操作神秘炸掉。

通用要求：日期填合理日期（不留空）、數值填業務合理值（非 0 或極值）、
格式欄位用合法格式、識別欄位用有語意的值、選項至少選一個、附件真的上傳。

> 各技術棧「壞值」的**具體後果與案例**（例如某型別的最小日期會讓某轉換拋例外）屬領域知識，見 `knowledge/`。

## 6. 大型功能：紅隊漏測複查（一次）

計畫（含覆蓋矩陣）完成後、交付執行**之前**，對「新互動＋新畫面」級別的功能，
由獨立視角只讀「矩陣 + diff + 驗收條件」（不執行），列出疑似漏測，QA Agent 依清單補 TC **一次**。
純樣式 / 文案 / 小改可略，但須註明「略過 + 理由」。

## 7. FAIL 前排除紀律（判 FAIL 之前必走，跨專案通用）

看到 FAIL **先別當成功能 bug**——多數「FAIL」其實是環境 / 探索 / 判讀問題。依序排除：

1. **服務存活先驗**：一次出現大量 FAIL，先確認前後端服務還在 listening（沒被中途關掉 / 沒崩）、
   測試環境前置（登入態 / test token / 必要資料）還在。環境塌了不是功能 bug。
2. **「操作沒生效」五排除**（click 後沒有預期的 API / 狀態變化時，逐一排除再判 FAIL）：
   - selector 點到的不是真元素（背景同名元素 / portal 在別處 / 被遮擋）→ 用唯一鍵 / role 重新定位
   - 事件沒被框架認得（受控元件只改 DOM 沒觸發框架事件）→ 見 `knowledge/` 受控元件段
   - 時序：操作後狀態還沒更新就驗 → 操作 → 等 → 再驗
   - 真實值與記憶 / 計畫不符（後端欄位名 / 端點名 / 按鈕文案對不上）→ **先 grep 原始碼確認真實值**，別盲試
   - 真的沒接（路由未 wire / 功能未實作）→ 這才可能是 bug
3. **業務語意三問（判「不給做」是 bug 還是 by design）**：當某操作「被擋下 / 不給送 / 看不到」時，
   先問：(a) 這是不是**刻意的卡控 / 權限設計**？(b) 後端回的是**業務碼（by design 擋）**還是真的炸（5xx）？
   (c) 換個合法前置 / 合法角色會不會就通？三問之後仍不通才登記 bug。

> 目的：避免把「環境塌 / 探索沒對準 / by-design 卡控」誤判成功能 FAIL，浪費往返並汙染報告可信度。
