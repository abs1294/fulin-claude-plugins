🧩 plugin-manager 使用教學

把你所有自製 Claude Code plugin 集中在一個地方管理，並能挑裝別人做的 plugin。

1. 這能幫我做什麼

如果你會自己寫 Claude Code 的 skill/plugin，平常會遇到這些麻煩——plugin-manager 幫你解決：

你的麻煩用哪個指令
自製 skill 散在各專案資料夾，想集中管理/plugin-manager:adopt 收編進來
改完 skill 要手動 git 推上去很煩/plugin-manager:publish 一鍵推
改了 skill 想發一個新版本給別的專案用/plugin-manager:update 升版號
每個專案要用的 plugin 不一樣，不想全開/setup-plugins 按專案挑
看到別人做的好 plugin，想推薦、之後挑裝（還能推廣給裝你 repo 的人）跟 Claude 說「推薦外部 plugin」+ /setup-plugins 挑裝

你不需要懂它背後怎麼運作就能用。想懂的話看最後一節「進階」。

2. 安裝（一次性）

在 Claude Code 輸入框依序貼這三行（Claude 不能代你執行 /plugin 指令，要你自己貼）：

/plugin marketplace add abs1294/fulin-claude-plugins
/plugin install plugin-manager@fulin-plugins
/reload-plugins
裝好後，/plugin-manager:adopt、:update、:publish 與 /setup-plugins 就能用了。

首次初始化（只做一次）

第一次用之前，要先建一份個人設定（告訴 plugin-manager 你的 monorepo 在哪）。跑一次 init 即可：

node "<你 clone 的 repo>/plugins/plugin-manager/scripts/init.js" <owner> <github-owner/repo>

例如：node ".../scripts/init.js" fulin abs1294/fulin-claude-plugins。monorepo 路徑會自動偵測（從腳本位置回推），不必手填；要指定可加第三個參數。它會在 ~/.claude/plugin-manager/ 建好 config.json 與 registry.json（已存在不覆蓋）。

為什麼要這步：所有指令都靠 ~/.claude/plugin-manager/config.json 找到你的 monorepo。這是個固定路徑的錨點——所以你之後在任何專案、任何 session 說「推薦外部 plugin」「adopt 某 skill」，Claude 都能從這個錨點查到該寫去哪，不必每次告訴它路徑。沒這份 config 的話指令會提示你先 init。

3. 指令速查表

指令什麼時候用怎麼下
/plugin-manager:adopt我在某專案寫好一個 skill，想收進集中管理/plugin-manager:adopt <skill名>
/plugin-manager:update改完某個 plugin，要發一個新版本/plugin-manager:update <plugin名> <patch｜minor｜major>
/plugin-manager:publish改了東西要推上 GitHub/plugin-manager:publish
/setup-plugins設定「這個專案」要啟用哪些 plugin/setup-plugins
/setup-plugins upgrade這專案的自製 plugin 想同步到最新版/setup-plugins upgrade
（對話）「推薦外部 plugin」把別人的 plugin 記進推薦清單（會推廣）直接跟 Claude 說「推薦外部 plugin …」
/plugin-manager:clone-env把整套環境複製到新機器/新專案跟 Claude 說「複製我的環境」「環境復現」

4. /plugin-manager:adopt — 收編我自己寫的 skill

什麼時候用：你在某個專案的 .claude/skills/ 裡寫了一個 skill，想把它收進集中管理（之後能發布、能在別的專案用）。

怎麼下

/plugin-manager:adopt my-tool

你會得到什麼

這個 skill 變成一個受管理的 plugin：之後可以 /plugin-manager:publish 發布、可以在任何專案 /setup-plugins 啟用。而且你在原專案那份照樣能用、改它就是改受管的那份（不會有兩份打架）。

adopt 只動你指定的那一個 skill，不會自動掃整個專案、不碰別的東西。若該專案是 git repo，它會提醒你把連結加進 .gitignore。

也能收編你自己寫的 agent

什麼時候用：你在專案 .claude/agents/ 裡寫了一組 agent，想集中管理 / 推廣。

跟 Claude 說「把這專案的 agents 納管成 <plugin名>」，它會跑：

node ".../scripts/adopt-agents.js" <plugin名> [專案路徑]

整個 agents 目錄一起收進 plugins/<plugin名>/agents/（agent 是單檔、不像 skill 是目錄，所以整包一起收，原位用整包 junction 指回）。一組 agent = 一個 plugin。裝了這 plugin 的人會自動載入這組 agent。

5. /plugin-manager:update — 改完要發新版

什麼時候用：你改了某個 plugin 的內容，想發一個新版本（這樣別的專案才抓得到更新）。

怎麼下

/plugin-manager:update git-commit patch
升版等級用於例
patch修字、修小 bug0.1.0 → 0.1.1
minor加功能（相容）0.1.0 → 0.2.0
major破壞性變更0.1.0 → 1.0.0

升完版記得 /plugin-manager:publish 推上去。

6. /plugin-manager:publish — 推上去

什麼時候用：你 adopt 了新 skill、或改了 / 升版了某個 plugin，要把這些推上 GitHub。

怎麼下

/plugin-manager:publish

你會得到什麼

Claude 會先給你看「這次改了什麼 + 建議的 commit 訊息」讓你確認，確認後一次 commit + push 整包。不管你這次改了 1 個還是 50 個 plugin，只推一次。

7. /setup-plugins — 設定這個專案要用哪些 plugin

什麼時候用：不同專案需要的 plugin 不一樣（前端專案要 figma、後端要 .NET 工具…）。這指令讓你按「當前專案」挑要啟用哪些，不必全部塞在一起。

怎麼下，以及我會怎麼問你

你只要輸入：

/setup-plugins

接著 Claude 會用選單問你（你只要點選）：

問題①：這個專案要套哪個 plugin 組合（profile）？
minimal（最精簡，只留全域常駐）
dotnet（.NET 後端：C# LSP + .NET 工具 + 資安審查）
frontend（Vue/TS 前端：TS LSP + Figma + 瀏覽器自動化）
supplier（供應商平台組合）
full（全開）
自訂（我逐一勾選）

這些 profile 定義在 ~/.claude/plugin-profiles.json，你可以自己編輯增減。

問題②（固定會問）：要不要從推薦清單挑「別人做的」外部 plugin 來裝？
（列出推薦清單裡的 plugin，連同用途說明，讓你勾選）
推薦多時：先問「要哪個面向（tag）」→ 只列該 tag 的，免一長串勾

推薦清單為空時，這題仍會出現，順便告訴你「可以跟我說『推薦外部 plugin …』來加」。

選完之後

Claude 把你的選擇寫進「這個專案目錄」的 .claude/settings.json（只影響這專案，不污染別的）。若有需要先裝的 plugin 或第三方來源，會給你要自己貼的 /plugin 指令。最後提醒你 /reload-plugins 生效。

子功能：/setup-plugins upgrade

當這專案啟用的自製 plugin 落後最新版時，這指令會偵測出來，並列出要貼的更新指令（先刷新、再 uninstall + install）。

8. 推薦別人做的 skill/plugin（會推廣）

什麼時候用：你在 GitHub 看到別人做的好用 plugin，想「精選收進你的推薦清單」——之後在 /setup-plugins 時能挑著裝，而且別人裝你的 repo 也會看到你推薦了什麼。

重點：只記下「來源 + 用途 + 標籤」，不會把別人的程式碼複製進你的 repo——別人的 plugin 永遠住在他們的 repo、跟著他們更新。你只是做一份「精選書籤」。
這份清單會被推廣：推薦寫進 plugins/plugin-manager/recommends.json（在 plugin 內、隨 repo 進 git）。/plugin-manager:publish 後，別人裝你的 repo 就看得到。
（對比：你自製 plugin 的本機狀態存在家目錄、不進 git、別人看不到。）

怎麼推薦一個外部 plugin

直接跟 Claude 說，例如：

推薦外部 plugin：dotnet-skills@dotnet-skills，
來源 winbond-db00/dotnet-skills，用途 .NET 後端技能包，tag backend,dotnet
  • 用途（note）必填——之後翻清單時靠它認出這 plugin 是幹嘛的。
  • tag 建議填——清單一多時，/setup-plugins 會先問你「要哪個面向」再只列該 tag 的，不必一長串勾選。
  • 要移除就說「移除推薦 dotnet-skills@dotnet-skills」。
登記 ≠ 安裝：推薦只是記一筆書籤（連那個 plugin 存不存在都不檢查），不會真的裝。真正安裝是下面那步、要你自己貼。

推薦後怎麼裝

記得先 /plugin-manager:publish 把清單推上去。之後跑 /setup-plugins，選完 profile 會問你「要不要從推薦清單挑幾個裝」（上面問題②，清單多會按 tag 分組）。勾了之後，Claude 給你要自己貼的安裝指令：

/plugin marketplace add <來源>
/plugin install <name@marketplace>
/reload-plugins

9. 把整套環境複製到新機器 / 新專案 — /plugin-manager:clone-env

什麼時候用：換了新電腦、或想在另一個專案/同事機器上，重現你慣用的整套 plugin 組合（含第三方的 figma/codex/claude-mem 與你自製的）。

這是個 skill——跟 Claude 說「複製我的環境到新機器」「擷取環境快照」「環境復現」就會觸發，跟 adopt 一樣不必記指令。Claude 幫你跑擷取/產生，最後安裝那幾步你自己貼。

📋 複製（在舊機器）

跟 Claude 說 「擷取我的環境快照」。Claude 會把環境清單存成 env-snapshot.json（寫進 plugin 內、隨 git），並提醒你 /plugin-manager:publish 推上去。

快照複製的是「清單」（不是程式碼）：

  • marketplaces：你加過的每個 marketplace 來源 repo
  • plugins：每個 plugin 的版本、scope、user 層是否啟用
  • projects：各專案（如 Supplier_Code、Ebilling）各自啟用了哪些 plugin（per-project 設定，只記專案名）

不複製：plugin 程式碼本身（靠重裝/clone）、API 憑證（新機各自登入）、registry 個人狀態。背後跑 export-env.js，你不必自己打。

📌 貼上（在新機器）

新機器照這個順序（前兩步是 bootstrap，要自己來；之後就自然語言）：

1
先裝管理器本身：/plugin marketplace add <你的 repo> → /plugin install plugin-manager@<marketplace>（這步把含快照的 plugin 帶進來）+ git clone 你的 monorepo + 跑一次 node init.js
2
跟 Claude 說 「復現環境」→ Claude 從帶過來的快照產生其餘 plugin 的安裝清單
3
逐行貼那串（Claude 代執行不了 /plugin）：
/plugin marketplace add <每個來源> → /plugin install <每個 plugin> → /reload-plugins
同機換專案：說「只列啟用中的復現指令」（--enabled-only），再配 /setup-plugins 寫進該專案——不必重裝、只調該專案啟用哪些。
第三方 plugin 能不能裝，看對方 marketplace：restore 只給你 marketplace add 指令，實際裝得起來要那個來源還在、且你有權限（有些要登入/授權）。你的自製 plugin（在你 repo）則一定裝得起來。

10. 常見問題

Q：為什麼 Claude 不直接幫我裝 plugin，要我自己貼指令？

/plugin install、/plugin marketplace add、/reload-plugins 都是 Claude Code 的「互動指令」，只能由你在輸入框親自打。plugin-manager 只負責「把設定寫好 + 把要貼的指令給你」。

Q：要更新一個已裝的 plugin，是不是 /plugin update？

沒有 /plugin update 這個指令。更新的正確做法：/plugin marketplace update fulin-plugins 刷新 → /plugin uninstall <name>@fulin-plugins 再 /plugin install <name>@fulin-plugins 重裝（或在 /plugin 介面開 Enable auto-update 自動更新）。

Q：adopt 後我在原專案改 skill，要重新 adopt 嗎？

不用。原專案那份和受管那份是同一份檔，改哪邊都一樣。改完直接 /plugin-manager:publish 推就好。

Q：我改完 skill，但別的專案沒拿到更新？

publish 只推上 GitHub，不會自動更新已裝的專案。要其他專案拿到新版，請在那專案刷新 + 重裝（見上一題），或開 auto-update。

Q：能分享給別人用嗎？

能。整套就是個普通 plugin，別人 install 後跑初始化，指定他自己的 repo/owner 即可（狀態存在各自家目錄，不會寫死你的）。

11. 進階：背後怎麼運作

以下是內部機制，純好奇才需要看，日常使用完全不必懂。

① 一個 monorepo 管全部（解決「一 plugin 一 repo」）

官方常見做法是「一個 plugin = 一個 git repo」，500 個 plugin 就 500 個 repo。plugin-manager 把所有自製 plugin 收進同一個 monorepo，永遠只有 1 個 repo 要維護。

fulin-claude-plugins/                  # 一個 git repo
├── .claude-plugin/marketplace.json    # 一個 marketplace
└── plugins/
    ├── plugin-manager/                # 管理器自己
    ├── git-commit/  delaylocal/  qa-webwright/
② move + symlink（為什麼「真身只有一份」）

adopt 時，把 skill 真身從專案 move 進 monorepo，原專案位置改成 symlink 指回去。

專案/.claude/skills/X（真身） → move → monorepo/plugins/X/...（真身搬這） → 原地建 → symlink 指回

所以你在原專案看到的是捷徑，改它就是改 monorepo 那份——同一份檔，不會不同步。

③ registry（集中狀態存哪）

狀態存在家目錄 ~/.claude/plugin-manager/（不在 plugin 內，所以 plugin 更新不會覆蓋）：

  • config.json：owner、monorepo 路徑、GitHub repo。
  • registry.json 的 selfMade：你自製 plugin 的版本/dirty（家目錄、不進 git、別人看不到）。

外部 plugin 的推薦清單不放這裡，而在 plugin 內的 recommends.json（進 git、會推廣）——這是「私人本機狀態」與「公開推薦」刻意分開。

④ 每個指令在背後實際做的事
  • adopt：move 真身 → 建 plugin.json → 更新 marketplace.json → 建 symlink → 更新 registry。
  • update：算新版號 → 寫回 plugin.json 的 version → 同步 registry。
  • publish：git add → commit（heredoc 防注入）→ push → 清掉 registry 的「未推」標記。
  • setup-plugins：讀 profile / recommends.json → 合併進專案 settings.json 的 enabledPlugins。

完整工作規範見 monorepo 的 plugins/plugin-manager/CONVENTIONS.md。
