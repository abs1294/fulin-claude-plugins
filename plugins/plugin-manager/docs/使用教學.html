🧩 plugin-manager 使用教學

1. 為什麼需要它

痛點：Claude Code 的官方做法常常變成「一個 plugin = 一個 git repo」。如果你寫了 500 個自製 plugin，就會有 500 個 repo —— 根本無法維護。

plugin-manager 的解法：所有自製 plugin 全部塞進同一個 monorepo 的 plugins/ 底下。永遠只有 1 個 git repo 要維護。新增第 501 個 plugin？還是同一個 repo。
你的痛點plugin-manager 怎麼解
一 plugin 一 repo，數量爆炸無法維護全部進一個 monorepo，永遠 1 個 repo
自製 skill 散在各專案資料夾，難集中adopt 把它收進 monorepo，原地留 symlink
改完要手動 git add/commit/push 很煩publish 一鍵推整個 monorepo
怕「同一個 skill 兩份不同步」move+symlink：真身永遠只有一份

2. 核心概念

① monorepo（所有自製 plugin 的家）

fulin-claude-plugins/                  # 一個 git repo
├── .claude-plugin/marketplace.json    # 一個 marketplace，列出所有 plugin
└── plugins/
    ├── plugin-manager/                # 管理器自己也是一個 plugin
    ├── git-commit/                    # 被 adopt 進來的
    ├── delaylocal/
    └── qa-webwright/

② move + symlink（真身只有一份，零不同步）

當你在某專案寫了自製 skill，adopt 會：

專案/.claude/skills/X（真身） → move → monorepo/plugins/X/skills/X（真身搬這） → 原地建 → symlink 指回 monorepo

從此真身住在 monorepo，你在原專案看到的是 symlink。你改原專案那份 = 改 monorepo 那份（同一份檔），不會有兩份打架。

③ registry（集中狀態）

放在你的家目錄 ~/.claude/plugin-manager/（不在 plugin 內，所以 plugin 更新不會覆蓋）。記錄你有哪些自製 plugin、版本、改了沒推。

3. 安裝

在 Claude Code 輸入框依序貼這三個指令（Claude 不能代你執行 /plugin，要你自己貼）：

/plugin marketplace add abs1294/fulin-claude-plugins
/plugin install plugin-manager@fulin-plugins
/reload-plugins
裝好後，/plugin-manager:adopt、/plugin-manager:update、/plugin-manager:publish 與 /setup-plugins（含 upgrade）就會出現。

4. /plugin-manager:adopt — 把專案的自製 skill 納管

把你在「當前專案資料夾 .claude/skills/」裡寫的自製 skill，收進 monorepo 集中管理。

用法

/plugin-manager:adopt <skill名稱>

例如你在某專案寫了 my-tool skill：

/plugin-manager:adopt my-tool

它會做什麼

1
把 專案/.claude/skills/my-tool 真身 move 進 monorepo/plugins/my-tool/skills/my-tool
2
幫新 plugin 建 plugin.json（version 0.1.0）
3
更新 monorepo 的 marketplace.json
4
原專案位置改成 symlink 指回 monorepo（在專案仍可用）
5
更新 registry，標記 dirty: true（已改未推）
「靠登記」原則：adopt 只動你明確指定的那個 skill，絕不自動掃整個專案、不碰別人的東西或專案機密。
專案是 git repo 時：它會提醒你把該 symlink 加進專案的 .gitignore，避免 symlink 污染專案版控。

5. /plugin-manager:update — 改現有 plugin + bump 版本

維護 monorepo 裡既有的自製 plugin。核心是 bump 版本號 —— Claude Code 靠 plugin.json 的 version 判斷有無更新，改了內容卻不 bump，其他專案就算刷新 marketplace 也抓不到新版。

用法

/plugin-manager:update <名稱> <patch|minor|major>
等級用於範例
patch修字、修 bug、小調整0.1.0 → 0.1.1
minor加功能、加指令但相容0.1.0 → 0.2.0
major破壞性變更0.1.0 → 1.0.0

它會把新版號寫回 plugin.json 並同步 registry（dirty: true）。

version 只設在 plugin.json，不在 marketplace.json 加 version 欄 —— 兩處都設會衝突，Claude Code 以 plugin.json 為準。

6. /plugin-manager:publish — 一鍵發布整個 monorepo

因為自製 skill 在 adopt 時真身已經搬進 monorepo，publish 不需要再「蒐集」，只做純 git：一次 commit + push 整個 repo。

用法

/plugin-manager:publish

流程

查待發布狀態→ 你確認 commit message→ git add -A→ commit→ push
關鍵價值：不管你這次改了 1 個還是 50 個 plugin，publish 只推一次。500 個 plugin 也只有一次 push。

7. /setup-plugins — 為這個專案選要啟用哪些 plugin

像 Python venv 那樣按專案隔離：讓你為「當前目錄」選一組要啟用的 plugin，寫進該目錄 .claude/settings.json 的 enabledPlugins（per-project 覆寫，不污染 user 全域）。避免把所有 plugin 都塞 user scope 浪費上下文。

背景：plugin「安裝」是全域一份（cache），但「啟用」是 per-scope 的。寫進「目錄/.claude/settings.json」的 enabledPlugins = project scope，只在該目錄樹生效、覆寫 user 全域。

用法

/setup-plugins

它會做什麼

1
讀 ~/.claude/plugin-profiles.json 的 profile 清單（如 minimal / dotnet / frontend / supplier / full）
2
用互動選單問你要套哪個 profile，或選「自訂（逐一勾選 plugin）」
3
把選定結果合併進當前目錄 .claude/settings.json 的 enabledPlugins（保留原有項目，不整檔覆蓋）
4
提示你 /reload-plugins（或重開 session）讓設定生效；若含未裝過的 plugin 或第三方 marketplace，提醒你先自貼對應 /plugin 指令
profile 定義在 ~/.claude/plugin-profiles.json：要新增/改組合，編輯該檔即可（每個 profile 是一組 enable 的 plugin 清單，名稱格式 name@marketplace）。

子流程：/setup-plugins upgrade — 專案同步到最新版

publish 只推 monorepo，不會自動更新已安裝該 plugin 的專案。/setup-plugins upgrade 幫你偵測「當前專案啟用的自製 plugin 是否落後 registry 最新版」，並列出要貼的更新指令。

它會做什麼

1
讀本專案 .claude/settings.json 的 enabledPlugins，篩出屬於 fulin-plugins 的自製 plugin
2
對照 registry 最新版，列出版本與 dirty 狀態（dirty = registry 最新版還沒 publish）
3
給你要自貼的更新指令鏈：/plugin marketplace update fulin-plugins → /plugin uninstall <name>@fulin-plugins → /plugin install <name>@fulin-plugins → /reload-plugins
沒有 /plugin update 這個指令：Claude Code 的 /plugin 沒有 update 子指令。更新一個已裝 plugin 的正解是先 /plugin marketplace update 刷新索引，再 uninstall + install（或在 /plugin 互動 UI 的 Marketplaces tab 開 Enable auto-update，下次啟動自動更新）。
只偵測不代更新：專案 settings 的 enabledPlugins 只存 name@marketplace、不帶版本號，且 /plugin 系列是互動指令 Claude 不能代執行。所以 upgrade 做「偵測 + 列出 + 給指令」，更新指令仍要你自己貼。

8. registry — 集中狀態

位置：~/.claude/plugin-manager/

檔案記什麼
config.json你的 owner 名、monorepo 路徑、GitHub repo、預設分支
registry.jsonselfMade：你所有自製 plugin + 版本 + 是否 dirty。externalCandidates：你手動加入的外部候選 plugin

範本帶在 plugin 內（templates/），首次使用時複製到家目錄。這設計讓 plugin 升級不會覆蓋你的實際狀態，也讓別人裝走後能初始化成自己的 repo。

9. 完整實例：git-commit 納管全紀錄

這是本系統真實跑過的一次完整操作（把供應商平台的 git-commit skill 納管）：

# 1. 在 Supplier_Code 工作目錄，把 git-commit skill 納管
/plugin-manager:adopt git-commit
  ✓ 已 move skill 真身進 monorepo
  ✓ 已建 plugin.json
  ✓ 已加入 marketplace.json
  ✓ 已建 symlink（junction）回原專案
  ✓ 已更新 registry

# 2. 結果：Supplier_Code 的 git-commit 變成 symlink
.claude/skills/git-commit -> .../fulin-claude-plugins/plugins/git-commit/skills/git-commit/

# 3. 一鍵發布
/plugin-manager:publish
  → 改動的 plugin: git-commit
  → git add -A → commit → push
  ✓ pushed to abs1294/fulin-claude-plugins
實證：納管後在 Supplier_Code 改 git-commit = 改 monorepo 那份（同一份真身），而 Supplier_Code 照常能用這個 skill。

10. 限制與注意事項

Claude 不能代執行 /plugin 系列：install、marketplace add、reload-plugins 都是互動指令，必須你自己在輸入框貼。plugin-manager 只負責「寫好設定 + git 操作 + 給你要貼的指令」。
Windows symlink 權限：adopt 用 junction → dir symlink fallback。若都失敗會回報「真身已在 monorepo，但原位置未連結」，需手動處理。
敏感內容：adopt 會把整個 skill 目錄搬進 monorepo 並可被 publish。納管前確認該 skill 沒有不該公開的東西（路徑、帳號、auth bypass 位置等）。若有敏感內容，確保 repo 是 private。
publish 不會自動更新已安裝的專案：推上去後，已裝該 plugin 的其他專案需各自刷新（/plugin marketplace update fulin-plugins → uninstall + install，或開 auto-update）。注意：Claude Code 沒有 /plugin update 子指令。

11. 常見問題

Q：第一次 publish 為什麼要手動破冰？

因為要用 plugin-manager 發布它自己，但它還沒裝起來（雞生蛋）。所以「管理器自己」的第一次發布是手動 git push，之後才用 /plugin-manager:publish。

Q：adopt 後我在原專案改 skill，要重新 adopt 嗎？

不用。move+symlink 之後，原專案那份是 symlink，改它就是改 monorepo 真身。直接 /plugin-manager:publish 推就好。

Q：能分享給別人用嗎？

能。整套（skills + registry 範本）就是一個普通 plugin。別人 install 後，跑初始化指定他自己的 monorepo / repo / owner（registry 存家目錄、不寫死你的）。

Q：registry 會被 plugin 更新弄丟嗎？

不會。registry 實體在 ~/.claude/plugin-manager/（家目錄），plugin 內只帶範本。更新 plugin 不碰你的家目錄狀態。
